### list vs 더블linkedlist
정리하면 AC 문제에서는 맨앞 숫자를 뺄때 단순 배열의 경우 맨 앞이 빠지면 나머지 m개가 m-1개 만큼 왼쪽으로 이동해야 해서 연산이 오래걸리기 때문에 안되고 
링크드리스트의 경우 수정 삭제가 느린 반면 더블링크드리스트는 앞뒤 노드가 연결되어 있어서 
수정이나 삭제가 o(1)로 빨라서 보통의 연결리스트는 더블링크드리스트로 되어있고 다만 여전히 조회의 경우 링크드나 더블링크드나 o(n)으로 느린건 마찬가지인거다


1. "배열(List)은 맨 앞 삭제 시 이동(Shift) 때문에 느리다?" → ✅ 정답 (True)
   맨 앞 하나를 빼면 뒤에 있는 수만 개가 우르르 앞으로 이사 와야 합니다.

비용: O(N) (치명적)

2. "더블 링크드 리스트는 앞뒤 연결이라 삭제가 $O(1)$이다?" → ✅ 정답 (True)
   앞(Head)이나 뒤(Tail) 삭제 시, 이사 갈 필요 없이 연결고리만 툭 끊으면 됩니다.

비용: O(1) (쾌적)

3. "보통 라이브러리의 연결 리스트는 더블이다?" → ✅ 정답 (True)
   파이썬 deque, 자바 LinkedList, C++ std::list 모두 더블 링크드 리스트입니다.

이유: 사용자가 앞(pop_front)을 원할지 뒤(pop_back)를 원할지 모르니 둘 다 빠르게 하려고요.

4. "조회(Search)는 둘 다 $O(N)$으로 느리다?" → ✅ 정답 (True)
   싱글이든 더블이든, "5000번째 놈 나와" 하면 0번부터 줄 타고 하나하나 넘어가야 합니다.

비용: O(N) (답답)


### 싱글 링크드 리스트 & 더블링크드 리스트 맨 뒤 삭제시
싱글 링크드 리스트
A -> B -> C -> D(Tail)
D 삭제시 C의 NEXT를 NULL로 바꿔줘야 하는데 앞으로 못가니 다시 A->B->C 처럼 처음부터 다시 탐색해서 C를 찾고 tail로 변경
더블 링크드 리스트
A <-> B <-> C <-> D(Tail) (양방향 연결)
D 삭제후에 C를 바로 찾고 next를 null로 변경해 tail로 설정
